IT, 특히 소프트웨어 개발 직종에서 '구현' 이라는 용어는 굉장히 자주 쓰인다. 대충 이해할 때는 '코딩' 이라는 말로 대체가 가능할 것 처럼 보이나 실제로는 좀 더 깊은 이야기일 수도 있기 때문에 정리해본다.

대충 사용하는 의미

앞서 이야기 했다시피, IT 개발 직종에서 '구현'의 의미는 이렇다.

구현하다 = 개발하다, 코딩하다

그렇다면 사전적 의미는 뭘까?

사전적 의미

구현(具現/具顯) (명사) 어떤 내용이 구체적인 사실로 나타나게 함.

하지만 이 글에서 사전적 의미는 크게 의미가 없다. 우리가 이해할 의미는 관용적인 의미다.

구세대(?)의 정의

굳이 구세대라고 표기한 것이 큰 의미를 가지는 것이 아니다. 특히 비하의 의미도 절대로 아니다. 그냥 시간적으로 과거의 시점을 다룬다고 생각하자.

과거에 주류로 쓰이던 언어들, 예를 들어 C나 C++ 같은 언어들은 헤더(Header)라 불리는 파일이 존재한다. 주로 .h, .hh, .hpp, .h++ 등의 확장자를 가지고 있던 파일인데, 이 헤더에는 주로 공공에서 쓰일 각종 상수나 함수 혹은 변수의 인터페이스(interface) 등을 정의하는 목적의 파일이다.

이와의 상대적인 파일로 .c 혹은 .cc, .cpp, .c++ 등의 확장자로 끝나는 파일이 있는데 이를 통틀어서 구현 파일이라고 부르기도 한다.

즉 이 둘을 나눠보면 헤더는 정의(definition)를 만들고 나머지 파일은 구현(implementation)을 만들다. 정의에는 말 그대로 구현체의 모습을 정의해서 어떻게 쓰는지 방법과 모양을 기술하는 반면 구현체는 정말 어떻게 동작하는 지를 실제로 만들어 놓은 것이다.

여기서 '구현' 이라는 의미가 정해진다. 정의 혹은 인터페이스는 실체가 없기 때문에 이것 만으론 아무 동작을 하지 않는다. '구현'은 실제 동작을 만들어 내는 것이다.

반대로 '정의'가 없는 구현도 존재할 수 있는데, 현대적인 용어로 private 구현이라고 볼 수 있다. '정의'가 없다면 어떻게 사용하는지 어떤 이름으로 부르는 지 알 수가 없기 때문이다. 이런 식으로 '정의'를 알려 주느냐 마느냐 의 차이로 구세대(?) 언어 용 라이브러리의 사용 방법을 간접적으로 알려 줄 수도 있었다.

현 세대(?)의 정의

현대적인 언어, 예를 들자면 Swift나 Python 같은 언어의 관점에서 보자면 '정의'와 '구현'이 하나의 파일에서 하나의 설계로 이뤄진다. 따라서 굳이 '구현'과 '정의'를 분리하지 않아도 되는 경우가 있다.

하지만 변수의 개념에서는 여전히 '정의'라는 개념이 쓰인다.

그리고 클로저(closure)와 같이 호출 시 별도의 구현부를 넘겨야 하는 경우가 있는 것 처럼 '구현'이라는 개념은 여전히 존재하면서도 이전과는 색다르게 사용되고 있다.

Swift의 프로토콜(protocol) 개념도 정의와 구현을 분리 시키는 중요한 요소다. 프로토콜은 정의만 존재하며 구현을 별도로 해 주어야 하는 개념이기 때문이다. 물론 구세대 적인 의미와는 좀 다른 의미의 구현인 점은 여전하다.

결론

그래서 사실 처음에 적은 대충 사용하는 의미가 틀린 게 아니다. 그저 '정의'와 만나게 되면 구분되는 것이 '구현'일 뿐이다.

즉 정의와 구현이 다르다는 것이 핵심이다. 이를 제대로 구분할 수만 있다면 어떻게 이해해도 틀린 말은 아닐지도 모른다.
