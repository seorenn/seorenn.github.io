Git

Seorenn Static Pages

Git은 현존하는 가장 대표적인 DVCS(Distributed Version Control System) - 분산형 버전 컨트롤 시스팀 - 로써 파일의 변화점을 추적 및 관리하는데 사용하는 도구입니다. 뭐 그냥 소스코드나 프로젝트 관리하는 도구라고 해도 완전히 틀린 말은 아니겠지요.

사용법 메모

설정

전역(global) 

git config --global user.name "your name"
git config --global user.email "your email address"

현재 프로젝트에만 설정하기(local) 

git config user.name "your name"
git config user.email "your email address"

저장소 만들기

git init

현재 디렉토리를 Git 저장소로 만듭니다. .git 를 비롯해 상황에 따라 여러 숨김 디렉토리가 생겨납니다.

보관용 저장소 만들기

git init --bare

보관용 저장소를 만듭니다. 직접적으로 코드를 추가하거나 커밋하는 용도라 아닐 원격으로 Git 저장소 자체를 푸시하거나 땡겨오기(pull)위한 용도입니다.

상황 보기

git status

가장 많이 사용하게 될 커맨드입니다. 현재 저장소의 변경되고 있는 사항을 간략하게 보여줍니다.

파일 스테이지(Stage) 하기

git add FILE_PATH

특정 변경된 파일을 스테이지 합니다. 만약 현재 저장소의 모든 변경된 파일을 스테이지 하려면 아래와 같이 할 수 있습니다. 

git add .

스테이지 취소하기

git reset FILE_PATH

파일명을 적지 않으면 모두 언스테이지 됩니다.

변경점만 스테이지 하기(Stage Patch)

git add -p

이렇게 하면 현재 저장소의 변경된 파일을 변경점 위주로 리뷰하며 스테이지할 패치를 고를 수 있습니다.

커밋(Commit)

git commit

저장소에 소속된 파일들이 스테이지 되지 않은 상태에서 아래처럼 하면 몽땅 스테이징 해서 커밋할 수도 있습니다. 

git commit -a

-m 옵션을 사용하면 커밋 메세지를 간단히 작성할 수 있습니다. 

git commit -m "Update several files"

커밋 메시지 수정하기

아래 명령으로 가장 최근의 커밋 메시지를 수정할 수 있습니다. 

git commit --amend

커밋 내역

git log

한 줄 단위로 좀 더 단순하게 보기 위해서 --oneline 옵션을 사용할 수 있습니다. 

git log --oneline

브랜치 그래프를 확인하기 위해 --graph 옵션도 사용할 수 있습니다. 

git log --oneline --graph

원격과 연결하기

git remote add origin somegit://foo.bar/address

origin 이라는 원격 저장소를 현재 저장소에 알려주는 커맨드입니다. 뒤의 주소는 연결하려는 원격 서비스(github)에서 제공하는 주소를 입력해야 합니다. 물론 로컬의 다른 bare 저장소 디렉토리를 지정하는 것도 가능합니다.

푸시(Push)

git push origin master

origin 이라는 이름의 원격 저장소에 master 라는 브랜치 이름으로 푸시합니다. -u 옵션으로 원격 브랜치와 로컬 브랜치를 연결시킬 수도 있습니다. 

git push -u origin master

이후 이제는 푸시만 하면 자동으로 연결된 원격 브랜치로 푸시가 됩니다. 

git push

풀(Pull)

git pull origin master

origin 이라는 이름의 원격 저장소의 master 라는 브랜치의 내용을 가져와서 로컬 master 브랜치에 합칩니다(merge). 아래의 두 커맨드를 하나로 합쳐놓은 것과 비슷합니다. 

git fetch origin master
git merge origin/master

비교(diff)

현재 수정된 내역을 보려면 diff 커맨드를 사용하면 됩니다. 

git diff

특정 브랜치와 비교하려면 브랜치 이름을 추가해주면 됩니다. 

git diff some_branch

브랜치 만들기

git checkout -B branch_name

현재 브랜치에서 가지를 쳐서 다른 브랜치를 만듭니다.

작업 브랜치 변경하기

git checkout branch_name

작업 브랜치를 특정 브랜치로 전환합니다. 다만 최신의 Git에서는 switch 커맨드도 사용할 수 있습니다. 

git switch branch_name

switch 커맨드의 용도는 브랜치를 전환한다는 의미에서 checkout 과 완전히 동일합니다.

브랜치 조회하기

git branch

위읙 경우는 로컬의 브랜치 리스트만 나타나며, 원격을 보기 위해서는 -r 옵션을 추가해야 합니다. 

git branch -r

브랜치 삭제하기

git branch -D branch_name

왜 만들 때는 checkout 을 쓰는데 지울 때는 branch 를 쓰는지는 모르겠습니다.

원격 브랜치를 삭제할 때는 아래 방법을 쓸 수 있습니다. 

git push origin --delete REMOTE_BRANCH_NAME

원격 브랜치 가져오기

원격에만 있는 특정 브랜치를 로컬에도 그대로 가져오기 위해서는 아래 커맨드를 이용합니다. 

git checkout -t origin/branch_name

이렇게하면 로컬에 branchname 브랜치가 생성되고 여기에 원격(origin)의 branchname 브랜치의 내용을 가져옵니다.

다른 브랜치의 내용 패치하기

다른 브랜치의 내용을 현재 브랜치로 몽땅 머지하려면 아래와 같이 할 수 있습니다. 

git merge another_branch_name

특정 파일만 복사해 오고 싶다면 checkout 을 사용할 수 있습니다. 

git checkout another_bracn file_path

특정 파일을 수정 단위로 패치하고 싶다면 -p 옵션을 사용할 수 있습니다. 

git checkout -p branch_name file_path

임시저장(stash)

편집 중이던 내용을 임시로 저장해놓고 수정 전 상태로 돌리고 싶다면 stash 명령을 사용할 수 있습니다. 

git stash

저장된 내용은 list 커맨드로 파악할 수 있습니다. 

git stash list

저장된 내용을 확인하려면 show 커맨드를 사용할 수 있습니다. 

git stash show stash_name

다시 수정하던 내용으로 복구 시키기 위해 pop 명령을 사용할 수 있습니다. 

git stash pop

pop 의 경우 저장된 내용이 몽땅 사라지는데, 만약 저장된 내용 자체를 그대로 남겨두고 싶다면 apply 를 사용해봅시다. 

git stash apply

작업 내역 취소하기

특정 파일의 작업 중인 내역을 취소하기 위해서도 checkout 커맨드를 사용할 수 있습니다. 

git checkout -- file_path

아래의 커맨드도 가능합니다. 

git checkout HEAD file_path

기타 스테이징된 것만 리셋하기 위해선 reset 커맨드를 쓸 수도 있습니다. 

git reset -- file_path

아예 수정된 내역 전체를 날리기 위해선 아래처럼 할 수도 있습니다. 

git reset --hard HEAD

백업 따윈 존재하지 않으니 조심해서 사용합시다.

커밋을 없애기 위해서도 reset 을 사용할 수 있습니다. 

git reset commit_id

다만 위 커맨드는 해당 커밋 ID 시점부터 시작해서 가장 최근의 커밋 내역까지 몽땅 날아가니 주의합시다.

커밋 합치기

예를 들어 가장 최근의 4개 커밋을 하나로 합치고 싶다면 rebase 커맨드를 이용할 수 있습니다. 

git rebase -i HEAD~4

-i 옵션은 인터랙티브(interactive) 하게 한다는 의미이며, HEAD~4 는 HEAD 에서 4단계의 커밋까지라는 범위를 지정하는 의미입니다.

이후 편집기가 뜨면서 합칠 커밋을 선택해야 합니다. 대표가 될 커밋은 pick 을 하고 합쳐질 커밋은 squash 로 표기하면 됩니다. 작업 시 하단에 도움말이 나타나므로 잘 읽어봅시다.

강제로 푸시하기

rebase 후 원격 저장소에 푸시하려 하면 오류가 날 수 있습니다. 이럴 때는 강제로 푸시하는 방법이 있습니다. 

git push --force branch_name

굉장히 자유롭게 쓸 수 있지만 중요한 master 브랜치를 강제로 푸시하는 등의 행위는 자제합시다. 강제 푸시는 여러분이 작업하기 위해 따로 생성한 브랜치에만 하는 것이 현명합니다.

파일 지우기

git rm FILE_PATH

로컬과 저장소 양쪽에서 파일이 지워집니다.

내 컴퓨터의 파일은 놔두고 저장소에서만 파일을 지우고 싶을 때는 아래 커맨드를 이용할 수 있습니다. 

git rm --cached PATH

이후 .gitignore 에 등록하지 않으면 untracked file 로 분류가 됩니다.

업데이트 무시하기

어떤 이유(?)로 특정 파일이 업데이트되어도 이를 커밋하지 않으려면 아래 커맨드를 이용할 수 있습니다. 

git update-index --assume-unchanged PATH

이제 해당 파일은 업데이트 되어도 status 에 나타나지 않습니다. 다만 이 방법은 영원하지 않습니다. 예를 들어 브랜치를 바꾸거나 pull 상황에 따라 다시 나타날 수 있으므로 주의합시다.

설정 분리하기

디렉토리별로 Git 설정을 분리하려면 우선 ~/.gitconfig 파일에 아래와 같은 식으로 설정할 수 있습니다. 

[user]
    email = your primary email address
    name = your name
[includeIf "gitdir:~/foo/bar/"]
    path = ~/.gitconfig-foo-bar

위의 설정의 경우 ~/foo/bar/ 라른 디렉토리 안에서는 ~/.gitconfig-foo-bar 라는 설정 파일을 읽어들이도록 동작합니다. 여기에 필요한 설정을 추가해주면 됩니다. 아럐는 ~/.gitconfig-foo-bar 파일의 예제입니다. 

[user]
    email = your secondary email address
    name = your name
[github]
    user = github user name

github 설정이 추가되어 있다는 점만 제외하면 지극히 평범한 Git 설정입니다.

페이저 비활성화

페이저(pager)란 less나 more 같은 도구로 터미널에서 로그나 변경사항을 볼 때 긴 내용을 편하게 볼 수 있게 도와줍니다. 하지만 사람의 눈이 아닌 다른 UNIX 유틸리티로 파이프로 넘기려고 할 때는 이런 페이저가 방해가 됩니다. 이 경우 --no-pager 옵션을 줘서 실행시킬 수 있습니다.

예를 들어 diff를 페이저 없이 쓰려면 아래와 같이 입력하면 됩니다. 

git --no-pager diff

이 옵션은 pager. 으로 시작되는 다양한 설정으로 설정도 가능합니다. 예를 들어 branch 명령에서 페이저를 영원히 비활성화 시키려면 아래의 커맨드를 이용할 수 있습니다. 

git config --global pager.branch false

이 외에도 pager.diff 등 다양한 설정이 있으므로 직접 필요한 것을 찾아봅시다. (물론 별로 설정할 일은 없을 것 같습니다😃)
